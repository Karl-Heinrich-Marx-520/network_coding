# 现代 C++ Boost.Asio 异步 Echo 服务器指南

## 简介
前文已经介绍了异步操作的 API，今天我们将编写一个简单的异步 Echo（回声）服务器。为了便于理解异步状态机的流转，本示例以“一问一答”的应答模式为主。

## Session 类
`Session` 类主要用于处理与单个客户端消息收发的会话。**【重构点】**：原代码使用裸数组 `char _data[1024]` 和危险的 `delete this`。在现代 C++ 中，我们使用 `std::array` 管理固定大小的缓冲区，并让 `Session` 继承 `std::enable_shared_from_this`，利用智能指针彻底接管生命周期。

```cpp
#include <iostream>
#include <array>
#include <memory>
#include <boost/asio.hpp>

namespace asio = boost::asio;
using tcp = asio::ip::tcp;

// 【现代 C++ 核心】：继承 enable_shared_from_this，允许在成员函数中安全地获取指向自身的 shared_ptr
class Session : public std::enable_shared_from_this<Session> {
public:
    explicit Session(tcp::socket socket) : _socket(std::move(socket)) {}

    void Start();

private:
    void DoRead();
    void DoWrite(std::size_t length);

    tcp::socket _socket;
    // 【现代 C++ 替代】：使用 std::array 替代裸露的 char _data[1024]，
    // 避免数组退化为指针，且自带边界检查（通过 .at()）和大小查询（.size()）。
    std::array<char, 1024> _data; 
};
```

### 启动与读取操作
在 `Start` 方法中，我们发起第一次异步读取。注意这里我们使用 `DoRead` 替代了原来的 `handle_read`，使语义更清晰。

```cpp
void Session::Start() {
    DoRead();
}

void Session::DoRead() {
    // 【生命周期魔法】：获取指向自身的 shared_ptr
    auto self = shared_from_this();

    _socket.async_read_some(
        asio::buffer(_data),
        // 【现代 C++ 替代】：使用 Lambda 替代 std::bind。
        // 按值捕获 self！只要这个 Lambda 还在 Asio 的事件队列中，
        // Session 对象的引用计数就至少为 1，绝对不会被提前析构！
        [this, self](const boost::system::error_code& ec, std::size_t bytes_transferred) {
            if (!ec) {
                // 简单打印接收到的数据（注意添加字符串结束符以防打印乱码）
                std::cout.write(_data.data(), bytes_transferred);
                std::cout << "\n";
                
                // 收到数据后，发起异步写入（Echo 回去）
                DoWrite(bytes_transferred);
            }
            // 【彻底消灭 delete this】：如果发生错误（如客户端断开），
            // 我们什么都不做。Lambda 执行完毕后，self 被销毁，
            // 引用计数归零，Session 会被安全、自动地释放。绝不会发生二次析构！
        }
    );
}
```

### 写入与循环读取操作
当写入完成后，我们再次发起读取，形成“应答式”的闭环。

```cpp
void Session::DoWrite(std::size_t length) {
    auto self = shared_from_this();

    asio::async_write(
        _socket, 
        asio::buffer(_data, length),
        [this, self](const boost::system::error_code& ec, std::size_t /*bytes_transferred*/) {
            if (!ec) {
                // 发送成功后，继续监听下一次客户端的输入
                DoRead();
            }
            // 同样，发生错误时不手动 delete，依靠智能指针自动回收
        }
    );
}
```

## Server 类
`Server` 类为服务器接收连接的管理类。**【重构点】**：原代码使用 `new Session` 传递裸指针，并在出错时 `delete`。现代 C++ 中，我们统一使用 `std::make_shared` 创建会话，将内存管理交给智能指针。

```cpp
class Server {
public:
    Server(asio::io_context& ioc, short port)
        : _ioc(ioc),
          _acceptor(ioc, tcp::endpoint(tcp::v4(), port)) {
        DoAccept();
    }

private:
    void DoAccept();

    asio::io_context& _ioc;
    tcp::acceptor _acceptor;
};

void Server::DoAccept() {
    _acceptor.async_accept(
        // 【现代 C++ 替代】：async_accept 可以直接接受一个 socket 对象，
        // 或者我们在这里创建一个临时的 socket 传给 acceptor。
        // 为了配合 shared_ptr，我们使用 acceptor 的异步接受并传入回调。
        [this](const boost::system::error_code& ec, tcp::socket socket) {
            if (!ec) {
                // 【现代 C++ 替代】：使用 std::make_shared 创建 Session，
                // 彻底告别 new Session 和手动 delete new_session。
                std::make_shared<Session>(std::move(socket))->Start();
            }
            // 无论成功与否，继续监听下一个连接
            DoAccept(); 
        }
    );
}
```
*注：原代码中 `start_accept` 每次 `new` 一个 Session 并把它的 socket 传给 acceptor，如果 accept 失败还要手动 `delete`。现代 Asio 的 `async_accept` 回调直接返回一个构造好的 `tcp::socket`，我们只需将其 `std::move` 给新创建的 `Session` 即可，代码极其简洁且异常安全。*

## 客户端
客户端的设计使用之前的同步模式即可。对于简单的测试客户端，同步阻塞模型更直观。**【重构点】**：消除 C 风格的 `char[]`、`strlen` 和 `cin.getline`，全面拥抱 `std::string` 和 `std::getline`。

```cpp
#include <iostream>
#include <string>
#include <boost/asio.hpp>

namespace asio = boost::asio;
using tcp = asio::ip::tcp;

int main() {
    try {
        asio::io_context ioc;
        
        // 构造 endpoint
        tcp::endpoint remote_ep(asio::ip::make_address("127.0.0.1"), 10086);
        tcp::socket sock(ioc);
        
        // 现代 C++ 错误处理：使用 error_code 避免抛出异常导致程序终止
        boost::system::error_code ec;
        sock.connect(remote_ep, ec);
        if (ec) {
            std::cerr << "Connect failed: " << ec.message() << "\n";
            return 1;
        }

        std::cout << "Enter message: ";
        // 【现代 C++ 替代】：使用 std::string 和 std::getline 替代 char[] 和 cin.getline
        std::string request;
        std::getline(std::cin, request);

        // 同步发送
        asio::write(sock, asio::buffer(request), ec);
        if (ec) {
            std::cerr << "Write failed: " << ec.message() << "\n";
            return 1;
        }

        // 同步接收（为了简单，这里复用 request 的长度作为预期接收长度）
        std::vector<char> reply(request.size());
        asio::read(sock, asio::buffer(reply), ec);
        if (ec) {
            std::cerr << "Read failed: " << ec.message() << "\n";
            return 1;
        }

        std::cout << "Reply is: ";
        std::cout.write(reply.data(), reply.size());
        std::cout << "\n";
    }
    catch (const std::exception& e) {
        std::cerr << "Exception: " << e.what() << "\n";
    }
    return 0;
}
```

运行服务器之后再运行客户端，输入字符串后，就可以收到服务器应答的字符串了。

## 隐患（原代码的致命缺陷 vs 现代 C++ 的降维打击）

原文档非常敏锐地指出了旧版 Asio 示例中的一个致命隐患：**二次析构（Double Free）**。

### 原代码的灾难场景：
在旧代码中，`Session` 是通过裸指针 `new` 出来的，并在回调中手动 `delete this`。
假设服务器正准备发送数据（调用 `async_write` 前），客户端突然断开连接。
1. TCP 层面触发读就绪（EOF），`handle_read` 被触发，发现 `error` 非 0，执行 `delete this`。**（第一次析构）**
2. 与此同时，底层写事件也可能触发错误，`handle_write` 被触发，发现 `error` 非 0，再次执行 `delete this`。**（第二次析构）**
3. 对同一块内存执行两次 `delete`，会导致程序直接崩溃（Core Dump），这在生产环境中是绝对不可接受的。

### 现代 C++ 的降维打击：
在我们的重构代码中，**彻底删除了所有的 `delete this` 和 `delete new_session`**。
我们利用了 `std::shared_ptr` 和 Lambda 按值捕获的特性：
1. 每次发起异步操作时，Lambda 都会按值捕获 `self`（即 `shared_from_this()`），使 `Session` 的引用计数 +1。
2. 当客户端断开时，无论是读回调还是写回调先触发，发现 `ec` 错误后，**直接 `return`（或什么都不做）**。
3. Lambda 执行结束，其内部的 `self` 被销毁，引用计数 -1。
4. 只有当**所有**关联的异步操作（读和写）的 Lambda 都执行完毕并销毁后，`Session` 的引用计数才会归零，触发**唯一一次**安全的析构。

**这就是现代 C++ RAII 和智能指针的魅力：用所有权语义（Ownership Semantics）将复杂的并发生命周期问题，降维成了简单的引用计数问题。**

## 总结
本文介绍了异步的应答服务器设计。虽然通过现代 C++ 重构后，代码变得异常安全且优雅，但这种“一问一答”的服务器并不会在实际生产环境中直接使用，主要有以下原因：

1. **非全双工**：该服务器的发送和接收强耦合（必须等写完才能读），无法做到应用层随时主动推送数据（未实现收发分离的全双工逻辑）。
2. **缺乏协议处理**：未处理 TCP 粘包/半包问题，未设计应用层协议（如 Header + Body），也未处理数据的序列化与反序列化。
3. **架构耦合**：网络 I/O 线程与业务逻辑处理线程没有解耦，如果业务逻辑耗时，会阻塞 Asio 的 `io_context`，导致整个服务器吞吐量断崖式下跌。

*注：原代码提到的“二次析构风险”已通过本文的 `std::shared_ptr` + `enable_shared_from_this` 方案彻底解决。*
