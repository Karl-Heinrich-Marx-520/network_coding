# 现代 C++ Asio 进阶：构建工业级全双工发送引擎

## 简介
前文我们通过 Lambda 闭包解决了 Session 的生命周期问题。现在，我们要实现真正的全双工通信：服务器可以随时调用 `Send()` 接口向客户端推送数据。
**核心挑战**：当业务层高频调用 `Send()` 时，如何保证数据**有序发送**且不发生**内存覆盖**？

**架构师原则**：
1. **消灭手工内存管理**：用 `std::string` / `std::vector` 替代 `new char[]` 和 `MsgNode`。
2. **Asio 原生并发控制**：摒弃在 Session 内部加 `std::mutex` 的低效做法，使用 Asio 的 `post` 机制将发送任务派发到 Socket 所在的执行上下文（Executor），实现**无锁串行化**。
3. **彻底告别 `std::bind`**：全面拥抱 Lambda 表达式。

## 一、 抛弃 `MsgNode`：让 RAII 接管内存

老文档中设计了 `MsgNode` 类来深拷贝数据：
```cpp
// ❌ 老文档的 C++98 做法：手动管理内存，极易泄漏
class MsgNode {
    char* _data;
    MsgNode(char* msg, int len) { _data = new char[len]; memcpy(_data, msg, len); }
    ~MsgNode() { delete[] _data; } 
};
```
**架构师点评**：在 C++17 中，手写这种只为了“深拷贝一段字节”的类是毫无意义的。`std::string`（或 `std::vector<char>`）天生具备深拷贝语义、自动内存管理，且完美兼容 `boost::asio::buffer`。

**现代 C++ 替代方案**：
我们直接使用 `std::deque<std::string>` 作为发送队列。每次调用 `Send` 时，数据会被自动深拷贝到 `std::string` 中，彻底杜绝了 Buffer 竞争。

## 二、 Session 类：无锁全双工发送引擎

我们将 Session 拆分为头文件和实现。这里重点展示核心的**发送状态机**和**无锁并发控制**。

### `Session.h` (接口与状态定义)

```cpp
#pragma once
#include <iostream>
#include <memory>
#include <string>
#include <deque>
#include <boost/asio.hpp>

class Server; // 前置声明

class Session : public std::enable_shared_from_this<Session> {
public:
    Session(boost::asio::ip::tcp::socket socket, std::weak_ptr<Server> server);
    ~Session();

    void Start();
    void Send(std::string msg); // 线程安全的全双工发送接口
    
    const std::string& GetUuid() const { return _uuid; }

private:
    void DoRead();
    void DoWrite();
    void Close();

    boost::asio::ip::tcp::socket _socket;
    std::weak_ptr<Server> _server;
    std::string _uuid;

    // 读 Buffer：串行执行，复用安全
    std::array<char, 4096> _recv_buffer{}; 
    
    // 写 Buffer 队列：全双工核心
    std::deque<std::string> _send_queue; 
    bool _is_writing = false; // 写状态机标志
};
```

### `Session.cpp` (核心逻辑实现)

```cpp
#include "Session.h"
#include "Server.h"
#include <boost/uuid/uuid.hpp>
#include <boost/uuid/uuid_generators.hpp>
#include <boost/uuid/uuid_io.hpp>

using boost::asio::ip::tcp;

Session::Session(tcp::socket socket, std::weak_ptr<Server> server)
    : _socket(std::move(socket)), _server(server) {
    _uuid = boost::uuids::to_string(boost::uuids::random_generator()());
}

Session::~Session() {
    std::cout << "[Session] Destroyed: " << _uuid << "\n";
}

void Session::Start() {
    DoRead();
}

// 【架构师核心】线程安全的 Send 接口
void Session::Send(std::string msg) {
    // 利用 Asio 的 post 机制，将发送逻辑派发到 socket 所在的 executor (通常是 io_context)
    // 这保证了 _send_queue 和 _is_writing 的访问永远是单线程串行的，彻底免去 std::mutex！
    boost::asio::post(_socket.get_executor(),
        [this, self = shared_from_this(), data = std::move(msg)]() mutable {
            bool write_in_progress = _is_writing;
            _send_queue.push_back(std::move(data)); // 数据深拷贝入队
            
            if (!write_in_progress) {
                _is_writing = true;
                DoWrite(); // 触发状态机
            }
        });
}

void Session::DoRead() {
    _socket.async_read_some(
        boost::asio::buffer(_recv_buffer),
        [this, self = shared_from_this()](boost::system::error_code ec, std::size_t bytes_transferred) {
            if (!ec) {
                // 使用 string_view 零拷贝处理收到的数据
                std::string_view msg(_recv_buffer.data(), bytes_transferred);
                std::cout << "[Session " << _uuid << "] Recv: " << msg << "\n";
                
                // Echo 测试：将收到的数据发回
                Send(std::string(msg)); 
                
                DoRead(); // 继续监听
            } else {
                Close();
            }
        });
}

void Session::DoWrite() { 
    // 绑定队列头部数据的内存。因为数据在 queue 里，内存绝对安全
    boost::asio::async_write(
        _socket,
        boost::asio::buffer(_send_queue.front()),
        [this, self = shared_from_this()](boost::system::error_code ec, std::size_t /*bytes_transferred*/) {
            if (!ec) {
                _send_queue.pop_front(); // 发送成功，出队
                
                if (!_send_queue.empty()) {
                    DoWrite(); // 队列还有数据，继续触发下一次写
                } else {
                    _is_writing = false; // 队列清空，重置状态机
                }
            } else {
                Close();
            }
        });
}

void Session::Close() {
    boost::system::error_code ec;
    _socket.close(ec);
    if (auto server = _server.lock()) {
        server->RemoveSession(_uuid);
    }
}
```

## 三、 深度剖析：为什么 `boost::asio::post` 优于 `std::mutex`？

老文档在 `Send` 函数中使用了 `std::lock_guard<std::mutex>` 来保护队列：
```cpp
// ❌ 老文档做法
void CSession::Send(char* msg, int max_length) {
    std::lock_guard<std::mutex> lock(_send_lock); // 加锁
    // ... 操作队列 ...
}
```

### 1. `std::mutex` 在 Asio 中的隐患
在 Asio 中，`Send` 可能被多个业务线程并发调用，而 `DoWrite` 的回调是在 `io_context` 的工作线程中执行的。如果使用 `std::mutex`：
*   **死锁风险**：如果在持有锁的情况下调用了 Asio 的同步操作，或者在回调中再次获取锁，极易引发死锁。
*   **上下文切换开销**：高频调用 `Send` 时，线程间争夺互斥锁会导致大量的上下文切换，严重拖慢吞吐量。

### 2. `boost::asio::post`：降维打击的无锁并发
现代 Asio 提供了 `post`（或 `dispatch`）机制。
```cpp
boost::asio::post(_socket.get_executor(), [this, ...] { ... });
```
**底层原理**：
`post` 会将你的 Lambda 任务打包，**安全地投递到 Socket 所在的 `io_context` 事件循环队列中**。
这意味着，无论外部有多少个线程同时调用 `Send()`，这些 Lambda 任务最终都会在 `io_context` 的线程中**排队、串行执行**。
*   **结果**：`_send_queue` 和 `_is_writing` 永远只会被同一个线程（或同一个 strand）访问。**我们彻底删除了 `std::mutex`，实现了真正的无锁编程（Lock-free at the application level）！**

## 四、 Server 类的微调

Server 类的逻辑与前文基本一致，只需确保提供线程安全的 `RemoveSession` 接口。由于 `Server` 本身可能面临多线程访问（例如外部调用踢人接口），`_sessions` 的 map 需要加锁，或者使用 Asio 的 `strand` 保护。这里为了简洁，展示基础的 `mutex` 保护（因为 Server 的管理操作频率远低于 Session 的收发）：

```cpp
class Server : public std::enable_shared_from_this<Server> {
public:
    // ... 构造函数与 Start 略 ...

    void RemoveSession(const std::string& uuid) {
        std::lock_guard<std::mutex> lock(_mutex);
        _sessions.erase(uuid);
    }

private:
    // ... DoAccept 略 ...
    std::unordered_map<std::string, std::shared_ptr<Session>> _sessions;
    std::mutex _mutex; // 仅用于保护 Server 级别的连接池
};
```

## 五、 总结与下一步演进

通过本文的重构，我们完成了一个**工业级的全双工发送引擎**：
1. **RAII 内存管理**：用 `std::string` 和 `std::deque` 彻底消灭了 `MsgNode` 和裸指针。
2. **无锁状态机**：利用 `boost::asio::post` 和 `_is_writing` 标志，实现了线程安全且高效的串行化发送。
3. **真正的闭包**：全面使用 Lambda 按值捕获 `self`，代码清爽且绝对安全。

**架构师的前瞻**：
目前的服务器虽然实现了全双工，但依然是 **“透传字节流”**。在真实的业务中，TCP 是流式协议，会发生**粘包和半包**现象（例如客户端发送了两个 100 字节的消息，服务器可能一次性收到 200 字节，或者分两次收到 50 字节和 150 字节）。

在下一篇文章中，我们将引入**应用层协议设计（Header + Body）**，结合 Asio 的 `async_read_until` 或自定义状态机，彻底攻克 TCP 粘包/半包这一网络编程的终极 Boss。
