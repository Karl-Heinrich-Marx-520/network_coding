# 现代 C++ Boost.Asio 异步网络编程指南

## 简介
本文介绍 Boost.Asio 的异步读写操作及注意事项。为保证知识便于吸收，本节仅介绍核心 API 的使用片段，下一节再编写完整的客户端和服务器程序。

首先，我们定义一个 `Session` 类。在现代 C++ 网络编程中，`Session` 代表服务器与单个客户端连接的上下文（Context）管理器。

```cpp
#include <memory>
#include <boost/asio.hpp>

namespace asio = boost::asio;
using tcp = asio::ip::tcp;

class Session : public std::enable_shared_from_this<Session> {
public:
    // 【现代 C++ 优化】：使用 std::shared_ptr 管理 Session 自身的生命周期
    // 因为异步回调可能在 Session 局部变量销毁后依然被触发
    static std::shared_ptr<Session> Create(asio::io_context& ioc) {
        return std::shared_ptr<Session>(new Session(ioc));
    }

    tcp::socket& Socket() { return _socket; }

    void Connect(const tcp::endpoint& ep) {
        // 【现代 C++ 优化】：使用 Lambda 替代早期的同步阻塞或复杂的回调绑定
        _socket.async_connect(ep, [self = shared_from_this()](const std::error_code& ec) {
            if (!ec) {
                // 连接成功逻辑
            } else {
                // 错误处理：现代 C++ 推荐使用 std::error_code 而非抛出异常或 perror
            }
        });
    }

private:
    // 构造函数私有化，强制通过 Create() 工厂方法创建，确保始终以 shared_ptr 管理
    explicit Session(asio::io_context& ioc) : _socket(ioc), _strand(ioc.get_executor()) {}

    tcp::socket _socket;
    // 【核心特性】：strand 用于在多线程 io_context 中保证当前 Session 的回调串行执行，避免加锁
    asio::strand<asio::io_context::executor_type> _strand; 
};
```

> **💡 架构师笔记：为什么不用 `std::shared_ptr<tcp::socket>`？**
> 早期教程喜欢把 `socket` 包装在 `shared_ptr` 里。但在现代 Asio 设计中，`socket` 的生命周期应该与 `Session` 严格绑定。我们直接将其作为 `Session` 的成员变量，并通过 `std::enable_shared_from_this` 管理 `Session` 本身，这样设计更内聚，且减少了智能指针的间接寻址开销。

## 异步写操作

在写操作前，我们先封装一个消息节点（Message Node），用来管理要发送的数据。

### 1. 告别裸指针：现代化的 MsgNode
**【过时写法】**：使用 `new char[]` 和手动 `delete[]`，极易引发内存泄漏。
**【现代替代】**：使用 `std::vector<char>` 或 `std::string`，利用 RAII 自动管理内存，并支持移动语义。

```cpp
class MsgNode {
public:
    // 发送构造函数：利用 C++11 移动语义，避免字符串拷贝
    explicit MsgNode(std::string msg) 
        : _data(std::move(msg)), _offset(0) {}

    // 接收构造函数：预分配指定大小的缓冲区
    explicit MsgNode(size_t capacity) 
        : _data(capacity, '\0'), _offset(0) {}

    // 获取底层数据指针，供 Asio buffer 使用
    char* Data() { return _data.data(); }
    const char* Data() const { return _data.data(); }
    
    size_t Capacity() const { return _data.size(); }
    size_t Offset() const { return _offset; }
    size_t Remaining() const { return _data.size() - _offset; }
    
    void Advance(size_t bytes) { _offset += bytes; }
    bool IsCompleted() const { return _offset >= _data.size(); }

private:
    std::string _data;   // 【现代 C++】：用 std::string 替代 char*，自带内存管理
    size_t _offset;      // 当前已处理的偏移量
};
```

### 2. 异步写操作与 Lambda 回调
接下来为 `Session` 添加异步写操作。我们将彻底抛弃晦涩的 `std::bind`，全面使用 **Lambda 表达式**。

```cpp
class Session : public std::enable_shared_from_this<Session> {
    // ... 前面的代码省略 ...
public:
    void WriteToSocket(std::string buf);

private:
    void DoWrite(); // 实际执行写入的内部方法
    std::queue<std::shared_ptr<MsgNode>> _send_queue;
    bool _is_writing = false; // 替代原 _send_pending，语义更清晰
};
```

#### 为什么需要发送队列？（解决数据交错问题）
`async_write_some` 不保证一次性发完数据（半包问题）。如果用户连续调用两次 `WriteToSocket`，而第一次还没发完，第二次的数据就会和第一次剩余的数据在底层 TCP 缓冲区交错，导致对端收到乱序数据（如 `"HelloHello World! World!"`）。
**解决方案**：引入应用层发送队列 `_send_queue`，保证严格 FIFO 顺序发送。

```cpp
void Session::WriteToSocket(std::string buf) {
    // 【现代 C++】：使用 std::make_shared 和 std::move 避免不必要的拷贝
    auto node = std::make_shared<MsgNode>(std::move(buf));
    
    // 【线程安全】：使用 asio::post 将任务投递到 strand 中执行
    // 这样即使多个线程同时调用 WriteToSocket，队列操作也是绝对线程安全的
    asio::post(_strand, [this, self = shared_from_this(), node = std::move(node)]() mutable {
        _send_queue.push(std::move(node));
        if (!_is_writing) {
            DoWrite();
        }
    });
}

void Session::DoWrite() {
    if (_send_queue.empty()) {
        _is_writing = false;
        return;
    }

    _is_writing = true;
    auto& front_node = _send_queue.front();

    // 【现代 C++】：使用 Lambda 捕获 this 和 shared_from_this() (self)
    // self 保证了在异步回调触发时，Session 对象依然存活
    _socket.async_write_some(
        asio::buffer(front_node->Data() + front_node->Offset(), front_node->Remaining()),
        [this, self = shared_from_this()](const std::error_code& ec, std::size_t bytes_transferred) {
            if (ec) {
                // 错误处理：记录日志并关闭连接
                return;
            }

            auto& node = _send_queue.front();
            node->Advance(bytes_transferred);

            if (!node->IsCompleted()) {
                // 没发完，继续发当前包的剩余部分
                DoWrite(); 
            } else {
                // 当前包发完，出队，尝试发下一个包
                _send_queue.pop();
                DoWrite();
            }
        }
    );
}
```

### 3. 终极简化：使用 `asio::async_write`
手动处理 `async_write_some` 的偏移量计算非常繁琐。Asio 提供了自由函数 `asio::async_write`，它内部会自动循环调用 `async_write_some`，**直到所有数据发送完毕或发生错误才会触发回调**。

```cpp
// 【强烈推荐】：使用 asio::async_write 替代手动循环 async_write_some
void Session::DoWriteAll() {
    if (_send_queue.empty()) {
        _is_writing = false;
        return;
    }

    _is_writing = true;
    auto& front_node = _send_queue.front();

    // 注意：这里调用的是全局的 asio::async_write，而不是 socket 的成员函数
    asio::async_write(
        _socket,
        asio::buffer(front_node->Data(), front_node->Capacity()),
        [this, self = shared_from_this()](const std::error_code& ec, std::size_t /*bytes_transferred*/) {
            if (ec) {
                // 处理错误...
                return;
            }
            // 只要没报错，async_write 保证数据已经 100% 发送完毕
            _send_queue.pop(); 
            DoWriteAll(); // 递归发送队列中的下一个包
        }
    );
}
```
> **⚠️ 警告**：`async_write_some` 和 `asio::async_write` **绝对不能**在同一个 Socket 上混合使用，否则会导致底层缓冲区状态混乱。

## 异步读操作

异步读操作与写操作类似，同样存在 `async_read_some`（可能只读一部分）和 `asio::async_read`（保证读满指定长度）的区别。

### 1. 基础读取：`async_read_some`
如果不考虑应用层粘包/半包问题，仅做固定字节或流式读取：

```cpp
class Session : public std::enable_shared_from_this<Session> {
    // ...
public:
    void ReadFromSocket();
private:
    void DoRead();
    std::shared_ptr<MsgNode> _recv_node;
};

void Session::ReadFromSocket() {
    // 预分配 1024 字节的接收缓冲区
    _recv_node = std::make_shared<MsgNode>(1024); 
    DoRead();
}

void Session::DoRead() {
    _socket.async_read_some(
        asio::buffer(_recv_node->Data() + _recv_node->Offset(), _recv_node->Remaining()),
        [this, self = shared_from_this()](const std::error_code& ec, std::size_t bytes_transferred) {
            if (ec) {
                // 客户端断开连接 (eof) 或其他网络错误
                return;
            }

            _recv_node->Advance(bytes_transferred);
            
            if (!_recv_node->IsCompleted()) {
                // 没读满 1024 字节，继续读
                DoRead(); 
            } else {
                // 读满 1024 字节，投递给业务逻辑层处理 (此处省略)
                // ProcessMessage(_recv_node);
                
                // 准备下一次读取
                _recv_node = std::make_shared<MsgNode>(1024);
                DoRead();
            }
        }
    );
}
```

### 2. 精确读取：`asio::async_read`
在实际协议开发中（如：前 4 字节是包头长度，后面是包体），我们需要**精确读取**指定长度的数据。此时应使用 `asio::async_read`。

```cpp
void Session::ReadExactFromSocket() {
    // 假设协议规定每次必须读满 1024 字节才算一个完整包
    _recv_node = std::make_shared<MsgNode>(1024); 
    
    // 使用全局 asio::async_read，它会一直读，直到填满 1024 字节或报错
    asio::async_read(
        _socket,
        asio::buffer(_recv_node->Data(), _recv_node->Capacity()),
        [this, self = shared_from_this()](const std::error_code& ec, std::size_t bytes_transferred) {
            if (ec) {
                // 错误处理...
                return;
            }
            
            // 只要没报错，这里 bytes_transferred 必然等于 1024
            // ProcessMessage(_recv_node);
            
            // 准备下一次读取
            ReadExactFromSocket(); 
        }
    );
}
```

> **💡 架构师总结**：
> 1. 永远优先使用 `asio::async_read` / `asio::async_write`，它们能帮你屏蔽底层的“半包”细节。
> 2. 只有在实现自定义协议解析（如读到 `\r\n` 就停止）时，才使用 `async_read_some` 配合 `asio::read_until`。
> 3. **生命周期管理**：在 Lambda 中捕获 `self = shared_from_this()` 是现代 Asio 编程的铁律，它能彻底杜绝“回调触发时对象已析构”导致的段错误（Segmentation Fault）。
